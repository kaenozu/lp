## 2. MVPの目的

### 検証したいこと

- 「雑学クイズ」というジャンルに、自分自身と身近な人が実際に兴趣を持つか
- 4択＋解説のシンプルな形式が、飽きずに続けられるか
- 問題の質と量で、利用継続率がどう変わるか

### 最小限成立する条件

- 4択クイズが正常に動作する
- 正解/不正解のフィードバックが即座に表示される
- 解説を読める
- カテゴリを選べる
- 難易度を選べる
- 問題データがローカルで完結する

### 初期版でやらないこと

- ユーザー投稿機能
- アカウント作成・ログイン
- ランキング
- オンライン同期
- 課金
- 広告表示
- AIによる自動出題
- コメント機能

## 3. 初期機能

### 必須機能

| # | 機能 | 説明 |
|---|---|---|
| 1 | 4択クイズ | 1問ずつ4つの選択肢から選ぶ |
| 2 | 解説表示 | 回答後に正解/不正解と解説を表示 |
| 3 | カテゴリ選択 | 出題するカテゴリを選ぶ（複数選択可） |
| 4 | 難易度選択 | easy / normal / hard から選ぶ |
| 5 | 正解/不正解表示 | 即座にフィードバック |
| 6 | 次の問題へ進む | ボタンタップで次の問題へ |
| 7 | 問題一覧/ランダム出題 | 全問からランダムに出題、またはカテゴリ内から |
| 8 | 問題データのローカル保持 | アプリ内にJSONで問題を内包 |

### 任意機能（MVP後）

- 今日の一問
- 連続正解数
- お気に入り
- 履歴
- 正答率
- 問題リクエスト

## 4. 問題データ設計

### JSONスキーマ

```json
{
  "id": "string（必須）",
  "question": "string（必須）",
  "choices": ["string", "string", "string", "string"]（必須、4件固定）,
  "answerIndex": 0（必須、0〜3の整数）,
  "explanation": "string（必須）",
  "category": "string（必須、カテゴリID）",
  "difficulty": "string（必須、easy/normal/hard）",
  "sourceNote": "string（必須、出典の簡易表記）",
  "createdAt": "string（必須、ISO 8601日付）",
  "updatedAt": "string（必須、ISO 8601日付）",
  "sourceUrl": "string（任意、出典URL）",
  "tags": ["string"]（任意、タグ配列）,
  "title": "string（任意、問題セット名）",
  "shortText": "string（任意、補足説明）",
  "detailText": "string（任意、詳細解説）"
}
```

### 仕様メモ

- `choices` は常に4件。3件や5件は認めない。
- `answerIndex` は0始まり。`choices[answerIndex]` が正解。
- `sourceNote` は必須。空文字は許容しない。
- `category` と `difficulty` は列挙値に限定する。
- 1ファイルに複数問題を配列で保持する。

## 8. 画面構成

### ホーム画面

- アプリ名・ロゴ
- カテゴリ選択（複数選択のチップ/ボタン）
- 難易度選択（easy / normal / hard）
- 「スタート」ボタン
- 設定バー（カテゴリ全選択 / 難易度リセット など）

### クイズ回答画面

- 問題番号 (1/10)
- カテゴリ表示
- 難易度表示
- 問題文
- 4択ボタン
- 選択すると即座に正解/不正解を表示
- 「解説を見る」ボタン → 解説画面へ遷移

### 解説画面

- 正解/不正解の判定表示
- 正解の選択肢をハイライト
- 解説文
- 出典注記（`sourceNote`）
- 「次の問題へ」ボタン
- 全問終了時は「結果を見る」または「ホームに戻る」

### 結果画面（任意・MVP後）

- 正解数
- 所要時間
- もう一度プレイ

### 設定 / このアプリについて（任意・MVP後）

- バージョン表示
- プライバシーポリシーへのリンク
- 利用規約へのリンク

## 9. ユーザーフロー

```
アプリを開く
  → ホーム画面
  → カテゴリを選択
  → 難易度を選択
  → 「スタート」をタップ
  → クイズ回答画面（1問目）
    → 選択肢をタップ
    → 正解/不正解を即座に表示
    → 「解説を見る」をタップ
    → 解説画面
      → 解説を読む
      → 「次の問題へ」をタップ
      → クイズ回答画面（2問目）
        → ...（以下繰り返し）
        → 全問終了
          → 結果画面またはホームに戻る
```

### 状態遷移の注意

- 回答後は「解説を見る」ボタンを表示。押すまで次に進めない（解説を必ず読む体験）。
- 「次の問題へ」は1問につき1回だけ表示される。

## 10. データ更新方針

### 初期版

- アプリ内にJSONファイルをバンドル
- 問題の追加・修正はアプリのバージョンアップで配信
- オフラインで完結する

### 将来の検討

- **Cloudflare Pages / GitHub Pages でJSON配信**: アプリ本体を更新せずに問題だけ差し替えられる
- **Firebase Remote Config**: 問題データの一部を動的に切り替えられるが、複雑になりすぎない範囲で
- **API化**: ユーザー投稿や問題リクエスト機能を追加する場合に検討

### 推奨方針

MVP時点ではアプリ内JSON固定でリリースし、問題追加のペースが週1〜月1程度になるまではこの方式でよい。問題数が100問を超えたり、頻繁に更新したくなったら外部配信に移行する。

## 11. 出典・正確性方針

### 出典の扱い

- すべての問題に `sourceNote` を記載する
- 参考にした資料がある場合は `sourceUrl` も記入する
- 参考元がWikipediaの場合は、「Wikipedia『〜』より」と简单的に記載
- 自ら作成した問題は「編集部作成」と記載

### Wikipedia等を参考にする場合

- Wikipediaの内容は誰でも編集できるため、絶対的な信頼性は前提としない
- 問題文と解説は独自編集し、必要に応じて複数資料でクロス確認する
- 誤りがあった場合に修正しやすいよう、`sourceUrl` を記録する

### 間違い報告を受け付けた場合

- お問い合わせフォームまたはメールで報告を受け付ける
- 受領後、出典と照らし合わせて確認
- 誤りが確定したら `updatedAt` を更新して修正、問題データをバージョンアップで配信
- 対応履歴はGitHub Issueまたは社内メモで管理

### 免責

- 「可能な限り正確性を追求していますが、誤りが含まれる場合があります」
- LP上、プライバシーポリシー、利用規約に記載
- 誤りによる損害については責任を負わない旨を明記

## 12. 広告・Analytics方針

### 現時点

- 広告は表示しない
- Analyticsは導入しない

### 将来的に検討する場合

| 項目 | 条件 | privacy.htmlに必要な記載 |
|---|---|---|
| AdMob広告 | 収益化が必要になった場合 | 広告配信事業者名、取得データ、利用目的 |
| Firebase Analytics | 利用状況の把握が必要になった場合 | サービス名、提供元、取得データ、利用目的、第三者提供の有無 |
| Crashlytics | クラッシュ調査が必要になった場合 | クラッシュログの取得と送信 |

### 現時点の記載方針

privacy.html には以下を明記する:
- 「現時点では広告を表示していません」
- 「今後、広告またはアクセス分析を導入する場合は、本ページにてお知らせします」

## 13. 法務・ストア注意点

### 子ども向けに見える場合の注意

- 「子どもでも楽しめる」という表現は可能だが、「対象年齢：3歳以上」などの断定は避ける
- 保護者の判断でお使いください、という但し書きを入れる
- 広告を入れる場合、COPPA等の児童保護規制に留意する

### 雑学の著作権

- 自ら編集・執筆した問題は著作権を保有
- 既存の書籍・記事から引用する場合は、フェアユースの範囲内で使用
- 引用元を明記することで著作権リスクを低減する

### 引用と出典表記

- 問題文に「〜によると」という表現を使う場合、出典を明記する
- 画像やイラストを使用する場合も同様に引用元を確認する
- 商用利用可能なソース（CC0, CC-BY等）を優先する

### 広告導入時の注意

- 子ども向けアプリとして分類される場合、広告ネットワークの児童保護設定を有効にする
- AdMobの場合、アプリの対象年齢設定に留意する

### 誤情報リスク

- 雑学クイズは娯楽目的であり、学術的な正確性を保証するものではない
- 誤りを発見した場合の修正フローを整備する（第11項参照）
